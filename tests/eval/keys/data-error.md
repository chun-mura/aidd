## シード欠陥

コアシード (バージョン間比較の分母): **DE1, DE2, DE3** (全件。昇格シードなし)。

| ID | 該当セクション | 対応観点 | 期待深刻度 | 欠陥 | 一致判定基準 |
|---|---|---|---|---|---|
| DE1 | データフロー | データフロー | high | 在庫数を DB と Redis の両方に保持し、`SyncBatch` と `ProductAdminService` がそれぞれ両方を直接更新している (持ち主不明の二重管理。片系更新失敗で恒久的な不整合) | 在庫数の二重管理・書き込み経路の複線化・不整合リスクを指摘していれば検出。解決策の形 (単一書き込み経路・TTL 等) は問わない |
| DE2 | エラー処理 | エラー処理 | high | 個別商品の書き込み失敗を catch してスキップし、失敗件数にも数えず exit code 0 で正常終了する (完全な silent failure。監視にも `SyncResult` にも現れない) | 個別失敗の握りつぶし・失敗が観測不能である旨を指摘していれば検出 |
| DE3 | エラー処理 | エラー処理 | mid | WMS タイムアウト時に在庫数 0 として同期する (欠測を偽データで埋める。在庫ありの商品が「在庫切れ」表示になる実害。ただし次回同期で回復するため mid) | タイムアウト時の 0 埋めが誤データを作る旨を指摘していれば検出。「前回値保持にすべき」等の代案の形は問わない |
