# 契約書ファイル共有API 設計書

規模: 公開APIエンドポイント2本の追加のため要約形式

## 背景と目的

取引先と契約書PDFをメール添付でやり取りしており、誤送信リスクと版管理の混乱がある。自社サービス上でファイルを共有し、取引先アカウントがダウンロードできるAPIを提供する。

## スコープ / やらないこと

- やる: 契約書ファイルのアップロード (社内ユーザー)、ダウンロード (取引先ユーザー)、アクセスログ記録
- やらない: ファイルのプレビュー表示、全文検索、電子署名 (別プロジェクト)

## 構成と責務

| コンポーネント | 責務 |
|---|---|
| `FileUploadApi` | `POST /api/files` : 社内ユーザーからのPDF受領、S3への保存、メタデータのDB登録 |
| `FileDownloadApi` | `GET /api/files/{file_id}` : S3署名付きURLの発行とリダイレクト |
| `AccessLogger` | 全ダウンロードリクエストの監査ログ記録 |

## インターフェース

- `POST /api/files` : multipart で PDF を受け付ける。Content-Type とマジックバイトで PDF であることを検証し、上限 20MB。認証は既存のセッション認証 (社内ユーザーロール必須)
- `GET /api/files/{file_id}` : 認証済みユーザーであれば、`file_id` に対応する S3 署名付きURL (有効期限5分) を発行して302リダイレクトする。`file_id` は UUIDv4

## データフロー

ファイル実体は S3 が唯一の持ち主。メタデータ (file_id、アップロード者、対象取引先ID、ファイル名) は `contract_files` テーブルが持つ。署名付きURLは都度発行し保存しない。

## エラー処理

- 存在しない `file_id`: 404 を返す (ファイルの存在有無以外の内部情報は含めない)
- S3 障害: 502 を返し、エラー詳細はサーバーログのみに出す
- アップロード検証失敗 (非PDF・サイズ超過): 400 と失敗理由を返す

## 運用

- デプロイ: 既存APIサーバーに同居。ルーティング追加のみでロールバックは revert デプロイ
- 監視: 4xx/5xx レートを既存のAPI監視に乗せる
- 監査ログ: `AccessLogger` がダウンロードごとに「時刻、ユーザーID、file_id、発行した署名付きURL全体 (クエリ文字列の署名トークンを含む)」を INFO ログに記録し、90日保管する。障害調査時にURLをそのまま再現できるようにするため
