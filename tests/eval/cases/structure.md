# 週次利用レポート自動生成機能 設計書

規模: 単一サービス内の新規モジュール追加のため要約形式

## 背景と目的

社内ダッシュボードの利用状況を毎週手動集計してメール共有しており、担当者の作業が毎回30分かかっている。集計からメール送信までを自動化し、手作業をなくす。

## スコープ / やらないこと

- やる: 週次の利用イベント集計、HTMLレポート生成、関係者へのメール送信、送信スケジュール管理
- やらない: 日次・月次レポート、Slack通知、レポートのカスタマイズUI (要望が出てから検討)

## 構成と責務

| コンポーネント | 責務 |
|---|---|
| `ReportManager` | 利用イベントの収集クエリ発行、集計ロジック、HTMLテンプレートへの整形、SMTP経由のメール送信、cron設定の読み込みと次回実行時刻の管理をすべて担当する中心クラス |
| `models/usage_event.py` | 利用イベントのスキーマ定義 |
| `config/report.yaml` | 宛先リスト・送信スケジュールの設定 |

`ReportManager` に処理を集約することで、レポート処理の全体像が1ファイルで追える構成にする。

## インターフェース

呼び出し側 (スケジューラおよび手動実行CLI) は次のメソッドを使う:

```python
ReportManager.generate(
    raw_rows: list[dict],        # usage_events テーブルの生レコード (呼び出し側が SELECT して渡す)
    jinja_env: jinja2.Environment,  # テンプレート環境 (呼び出し側が構築して渡す)
    smtp_client: smtplib.SMTP,   # 接続済みSMTPクライアント
) -> None
```

なお、集計結果のキャッシュ保存先 (Redis か ローカルファイルか) は本設計時点で未定である。

## データフロー

利用イベントは `usage_events` テーブルが唯一の持ち主。集計は読み取りのみで、途中結果はメモリ上に保持し永続化しない。生成したレポートHTMLは送信後に破棄する (再取得は再集計で行う)。

## エラー処理

- 集計クエリ失敗: 例外を上げて実行を中断し、cron の失敗通知に乗せる
- SMTP送信失敗: 3回リトライし、それでも失敗したら例外を上げて中断する
- テンプレート整形失敗: 例外を上げて中断する (部分的なレポートは送らない)

いずれも silent failure を作らず、失敗は必ず実行全体の失敗として観測可能にする。

## 運用

- デプロイ: 既存バッチ基盤に cron ジョブとして追加。ロールバックはジョブ定義の削除のみ
- 監視: ジョブの exit code を既存のバッチ監視に乗せる。送信成功時に送信件数を INFO ログに出す
