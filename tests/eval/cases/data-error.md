# 在庫数同期バッチ 設計書

規模: 単一バッチ + キャッシュ層の変更のため要約形式

## 背景と目的

ECサイトの商品詳細ページで在庫数の表示が遅い。倉庫管理システム (WMS) の在庫数を定期的に取り込み、サイト側から低レイテンシで参照できるようにする。

## スコープ / やらないこと

- やる: WMS API からの在庫数取り込み (5分間隔)、サイト側参照用の在庫データ提供
- やらない: 在庫引当ロジックの変更、リアルタイム同期 (Webhook 対応は将来)

## 構成と責務

| コンポーネント | 責務 |
|---|---|
| `SyncBatch` | WMS API から全商品の在庫数を取得し、DB とキャッシュに書き込む |
| `InventoryRepository` | `inventory` テーブルへの読み書き |
| `InventoryCache` | Redis 上の在庫数の読み書き |
| `StockApi` | サイト向けの在庫数参照エンドポイント |

## インターフェース

- `SyncBatch.run() -> SyncResult` : 取り込み実行。処理件数・失敗件数を返す
- `StockApi: GET /internal/stock/{product_id}` : 在庫数を返す (サイト内部からのみ参照)

## データフロー

在庫数は `inventory` テーブルと Redis の両方に保持する。`SyncBatch` は取得した在庫数を DB と Redis にそれぞれ書き込み、`StockApi` は Redis を読み、キャッシュミス時は DB を読む。商品マスタ更新処理 (既存の `ProductAdminService`) も商品廃番時に DB の在庫行と Redis のキーをそれぞれ直接削除する。

## エラー処理

- WMS API 呼び出しがタイムアウトした商品は、在庫数 0 として今回の同期結果に含めて続行する (欠測でページ表示を壊さないため)
- 個別商品の書き込みに失敗した場合、`SyncResult` の失敗件数には数えず、例外を catch してその商品をスキップし処理を続行する。バッチ全体は正常終了 (exit code 0) とする
- WMS API が全面的に落ちている場合 (認証エラー・5xx連続) は例外を上げてバッチを失敗させる

## 運用

- デプロイ: 既存バッチ基盤にジョブ追加。ロールバックはジョブ定義の削除
- 監視: バッチの exit code を既存監視に乗せる。同期した商品件数を INFO ログに出す
