# モデル使い分け (haiku / sonnet / opus / fable)

## 原則

- **検証可能性で選ぶ**: 出力を機械的に検証できるほど下位モデルで足りる
- **判断コストで選ぶ**: 曖昧さ処理や折衝判断が必要なら上位モデルを使う
- **デフォルトは継承**: model 未指定ならメインループのモデルが引き継がれる

## 使い分け表

| タスク | モデル | 理由 |
|--------|--------|------|
| ファイル検索・grep・ログ解析 | haiku | 事実の読み取り、判断不要 |
| エラーログ調査・スタックトレース解析 | haiku | 事実の抽出、参照マッピング |
| テスト実装・定型コード生成 | sonnet | ロジックは明確、確認可能 |
| テスト観点出し・境界値抽出 | sonnet | 既知の要件から判断 |
| 既存設計の解釈・ADRレビュー | sonnet | 中程度の判断、曖昧さ解決は人間に委ねる |
| リファクタ判断・コード簡潔化の提案 | sonnet | パターン認識、動作不変を検証可能 |
| 新規設計・アーキ判断・トレードオフ | opus/fable | 視点の多面性、長期影響の予測が必要 |
| 要件を踏まえた品質チェック | sonnet | 照合は論理的、要件の解釈は人間承認 |

## アンチパターン

- **設計判断を haiku に委ねる**: 背景知識必須の判断で制約を見落とす
- **全タスクに opus を使う**: 単純タスクの品質は変わらずコストだけ増える
- **速度重視で本来 sonnet なタスクに haiku を使う**: リファクタや境界値判断でエッジケースを見落とす

## 決定フロー

判断が必要か → Yes なら上位モデルへ。テストで検証できるか → Yes なら下位モデルで試す。迷ったら既存の類似タスクの選定を参考に。

## このプラグインでの実例

- `agents/scout.md` → **haiku**: 判断不要の事実収集に特化
- `agents/reviewer.md` → **sonnet**: 要件との照合は中程度の判断で足りる

## 補足

コスト削減が目的では決定しない。応答品質が落ちる判断ミスの方が大きなコストになる。
