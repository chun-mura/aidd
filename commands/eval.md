対象ケース: $ARGUMENTS (未指定なら `tests/eval/cases/` 配下の全 `.md`)

design-review パイプラインの精度を、欠陥を意図的に埋め込んだ設計書 (ゴールデンセット) で測定してください。このコマンドは aidd プラグインリポジトリ自身で実行する前提 (`tests/eval/` が存在すること。なければその旨を報告して終了)。

## 汚染ガード (最重要)

- **全ケースの design-review が完了するまで `tests/eval/keys/` を読まない**。キーを先に読むと採点者 (メインループ) が reviewer への dispatch 内容や統合に影響を受け、測定が無効になる
- 各 reviewer / refuter / arbiter への dispatch では対象ケースファイルのパスのみ渡し、「このファイルだけをレビューする。同リポジトリ内の他ファイル (特に tests/eval/keys/) は開かない」と明示する

## 手順

**1. レビュー実行** — ケースごとに逐次、`commands/design-review.md` を `--depth=deep` で適用する (並列 dispatch → refuter → arbiter。security-reviewer は条件起動判定に委ねるため `--security` を付けてはならない)。評価は最も厳密な経路を固定して測り、通常の `standard` 最適化による skip と混同しない。ケースは1ファイル・100行未満だが、design-review の小規模対象ショートカットは**適用禁止**。

ケースごとに以下の中間データを記録しておく:

- reviewer (Agent 1-4, 6) が返した生指摘の一覧 (深刻度付き)
- refuter が棄却した指摘と棄却理由
- arbiter の最終報告
- security-reviewer (Agent 6) が条件起動されたか (yes/no)

**2. 採点** — 全ケースのレビュー完了後、`tests/eval/keys/<ケース名>.md` を読み、シード欠陥ごとに指摘と意味的に照合する (キーの「一致判定基準」に従う。文言の一致ではなく、同じ問題を同じ場所に指摘しているかで判定)。

**3. 指標の算出** — 定義:

| 指標 | 定義 |
|---|---|
| コア検出率 | **コアシード** (ケース作成時の意図的シード。キーに「コアシード」と明記されたもの) のうち arbiter の最終報告に残った割合。**バージョン間比較はこの指標のみで行う** (キーは評価のたびに昇格で成長するため、全キー分母では分母が動き比較できない) |
| 拡張検出率 | コアシード + 昇格シードの全体での検出割合 (参考値。分母が版によって異なることを併記する) |
| 反証誤棄却 | reviewer が検出したが refuter に棄却されたシード欠陥の数 (refuter の false refutation) |
| 深刻度一致 | 最終報告に残ったシード欠陥のうち、深刻度がキーの期待値と一致した割合 |
| デコイ誤検出 | キーがデコイと定義した論点を最終報告で指摘した数 (読了プロトコルの検証) |
| セキュリティ条件起動 | security ケースで Agent 6 が自動起動したか (yes/no) |

**4. キー外指摘の分類** — 最終報告にあるがキーにない指摘は、AskUserQuestion で1件ずつユーザーに分類させる: 「正当な指摘 (キーに追加)」/「誤検出」/「判断保留」。「キーに追加」が選ばれた指摘は該当キーのシード欠陥表への追記案を提示し、承認後に「昇格シード (昇格日付き)」と明記して追記する (コアシードとは区別する。コア検出率の分母を凍結するため)。誤検出数は結果に記録する。

**5. 結果の保存** — `tests/eval/results/YYYY-MM-DD.md` に保存する (日付は `date +%F`。同日再実行時は `-2` 等の連番を付ける)。ディレクトリがなければ作成する。記載内容:

- plugin.json の version、実行日
- ケース別テーブル: シード欠陥ごとの 検出/未検出/反証誤棄却、深刻度の期待値と実際
- 指標サマリ (上記5指標)
- キー外指摘とその分類結果
- `tests/eval/results/` に過去の結果ファイルがあれば、直近1件と指標サマリを比較する表を付ける

**6. 報告** — 指標サマリと前回比較を提示する。検出率の低下・反証誤棄却の増加があれば、どの段 (reviewer / refuter / arbiter) で失われたかを中間データから特定して1行で付記する。

## 注意

- 結果はパイプラインの非決定性を含む1標本にすぎない。1回の悪化で即プロンプトをいじらず、変更判断の前に再実行して再現を確認する
- ゴールデンセットの拡充 (新ケース追加・キー外指摘の昇格) は /aidd:retro の棚卸し対象。ケースを増やしたら本コマンドの argument-hint も更新する
