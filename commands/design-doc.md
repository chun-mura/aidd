対象要件: $ARGUMENTS

対象プロジェクトの設計書を作成し、`docs/design/YYYY-MM-DD-<slug>.md` に保存してください (日付は `date +%F`、slug は英語ケバブケース)。ディレクトリがなければ作成する。

**書く前に**: 設計を左右する不明点 (スコープ境界、優先するトレードオフ、既存資産の扱い等) があれば AskUserQuestion で確認する。要件自体が固まっていないなら先に superpowers:brainstorming で掘り下げる。関連する既存コードがあれば読み、現状を踏まえて書く。

**構成** (aidd:design-review の6観点と1対1対応。レビューで照合される前提で書く):

1. **背景と目的**: なぜやるか。1-3行
2. **スコープ / やらないこと**: 今回やる範囲と、意図的に含めない拡張 (YAGNI 観点の照合対象)
3. **構成と責務**: コンポーネント一覧と各1責務。境界を明示
4. **インターフェース**: 外部から見た使い方。内部を読まずに分かる形で
5. **データフロー**: 状態の持ち主、データの流れ。二重管理がないことを示す
6. **エラー処理**: 失敗経路の列挙と各対処。silent failure を作らない方針
7. **運用**: デプロイ・ロールバック・監視の考慮

**ボリュームは規模で自動判断**:
- 数ファイルの変更・単一モジュール → 各節1-3行、全体1ページ以内
- 新規サービス・複数コンポーネント・外部連携あり → フル記述 (図・テーブル可)

判断に使った規模認識を冒頭に1行書く (例: 「規模: 単一モジュール変更のため要約形式」)。

**保存後**: パスを報告し、`/aidd:design-review <パス>` の実行を提案する。加えて、設計から実装規模を見積もり (目安: 変更ファイル5超 または 独立してマージ可能な単位が2つ以上)、閾値を超える場合は `/aidd:issue-split <パス>` の実行も提案する (レビュー単位が小さいほどレビュー精度が上がるため、PRが大きくなる前に分割する)。

**frontmatter**: 保存する設計書ファイルの先頭に以下を付与する:

```yaml
---
status: draft
---
```

`status` は `draft` (未実装) / `implemented` (実装済み) / `superseded` (後継設計に置き換え) のいずれか。新規作成時は常に `draft`。既存設計書を更新する場合は既存の `status` 値を保持する (design-sync が更新対象)。
