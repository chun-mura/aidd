argument-hint: [設計ファイルのパス or 設計の要約] [--verify-sources]
---

対象: $ARGUMENTS

`$ARGUMENTS` に `--verify-sources` が含まれる場合はフラグとして解釈し、各 agent へ渡す対象からは除去すること。

以下の3グループを **単一メッセージで aidd:reviewer agent に並列 dispatch** し、メインコンテキストで統合してください。各 agent には対象 (ファイルパス or 要約) と担当観点のみ渡すこと。対象が小さい (1ファイル・100行未満の要約) 場合のみ、dispatch せずメインで直接レビューしてよい。

対象プロジェクトに `.aidd/design-perspectives.md` が存在する場合、その内容を担当観点として渡す **Agent 4 — プロジェクト固有観点** を同じメッセージで並列 dispatch に追加する (標準6観点は常に維持し、置き換えない)。ファイルがなければ Agent 4 は起動しない。

**Agent 1 — 構造:**
1. 責務分割: 1コンポーネント1責務になっているか。境界が曖昧な箇所はないか
2. インターフェース: 内部を読まずに使い方が分かるか。変更が波及しない構造か

**Agent 2 — データとエラー:**
3. データフロー: 状態の持ち主が明確か。同じデータの二重管理はないか
4. エラー処理: 失敗経路が設計されているか。silent failure の余地はないか

**Agent 3 — 実用性:**
5. YAGNI: 今必要ない拡張性・抽象化が入っていないか
6. 運用: デプロイ・ロールバック・監視の考慮があるか

**Agent 4 — プロジェクト固有 (存在する場合のみ):**
`.aidd/design-perspectives.md` の内容をそのまま観点として使う。

**Agent 5 — 外部情報源検証 (`--verify-sources` 指定時のみ):**
**aidd:source-verifier agent** を同じメッセージの並列 dispatch に追加する。渡すのは対象のみ (検証可能な主張の抽出は verifier 自身が行う)。フラグがなければ起動しない (Web 検証はコスト高のため opt-in)。

Agent 1-4 への指示: 観点ごとに「問題なし」か「指摘 (深刻度: high/mid/low)」を返す。指摘には必ず具体的な代案を付ける。Agent 5 の出力形式は agent 定義側 (確認済み/反証あり/未確認) に従う。

統合時: 起動した全 agent の結果が揃ったら、対象と全指摘を **aidd:design-arbiter agent (opus)** に渡して裁定させる。arbiter が矛盾の裁定・深刻度の再ランク付け・「最優先で直すべき1点」の選定を行う。Agent 5 が起動している場合、arbiter への指示に「source-verifier の『反証あり』のみ深刻度付き指摘として扱い、『未確認』は指摘に含めず情報として併記する」を加える。メインは arbiter の結果をそのまま報告する (メインループのモデルに依存させないため、統合判断を自分で行わないこと)。重大な設計判断でセカンドオピニオンが欲しい場合は codex:rescue に同じ観点を投げて突き合わせる選択肢がある。
