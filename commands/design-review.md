対象: $ARGUMENTS

`$ARGUMENTS` に `--verify-sources` / `--security` / `--no-security` が含まれる場合はフラグとして解釈し、各 agent へ渡す対象からは除去すること。

レビュー開始時に対象プロジェクトの `.aidd/review-dismissed.md` が存在する場合は読み込む。対象ドキュメントが削除済み・大幅改訂済みのエントリは棄却一覧から除外してよい。残った一覧と、`$ARGUMENTS` に含まれる棄却済み指摘 (過去ラウンドでユーザー承認により棄却されたもの) を各 reviewer agent に渡し、「同一内容の指摘を再報告しない」よう指示すること。

以下の3グループを **単一メッセージで aidd:reviewer agent に並列 dispatch** し、メインコンテキストで統合してください。各 agent には対象 (ファイルパス or 要約) と担当観点のみ渡すこと。対象が小さい (1ファイル・100行未満の要約) 場合のみ、dispatch せずメインで直接レビューしてよい。ただし `--verify-sources` 指定時はこのショートカットを適用せず、通常の並列 dispatch 経路に乗せること (Agent 5 を確実に起動するため)。

対象が大きい場合 (目安: 5ファイル超 または 合計3000行超)、1つの agent に全ファイルを渡さず、観点グループごとに対象ファイルを分割して複数 agent を起動すること (1 agent の担当量を全文精読できる範囲に保つ)。ただしファイル横断の整合性を見る観点 (二重管理・責務境界の重複など) を担当する agent には対象全体の一覧を渡す。

対象プロジェクトに `.aidd/design-perspectives.md` が存在する場合、その内容を担当観点として渡す **Agent 4 — プロジェクト固有観点** を同じメッセージで並列 dispatch に追加する (標準6観点は常に維持し、置き換えない)。ファイルがなければ Agent 4 は起動しない。セキュリティ観点を常設したいプロジェクトは、このファイルに追加する (出発点として `templates/design-perspectives.md.template` をコピーできる)。

**Agent 1 — 構造:**
1. 責務分割: 1コンポーネント1責務になっているか。境界が曖昧な箇所はないか
2. インターフェース: 内部を読まずに使い方が分かるか。変更が波及しない構造か

**Agent 2 — データとエラー:**
3. データフロー: 状態の持ち主が明確か。同じデータの二重管理はないか
4. エラー処理: 失敗経路が設計されているか。silent failure の余地はないか

**Agent 3 — 実用性:**
5. YAGNI: 今必要ない拡張性・抽象化が入っていないか
6. 運用: デプロイ・ロールバック・監視、ログ設計・メトリクス設計 (何を観測できれば障害調査・性能分析が成立するか) の考慮があるか

**Agent 4 — プロジェクト固有 (存在する場合のみ):**
`.aidd/design-perspectives.md` の内容をそのまま観点として使う。

**Agent 5 — 外部情報源検証 (`--verify-sources` 指定時のみ):**
**aidd:source-verifier agent** を同じメッセージの並列 dispatch に追加する。渡すのは対象のみ (検証可能な主張の抽出は verifier 自身が行う)。フラグがなければ起動しない (Web 検証はコスト高のため opt-in)。

**Agent 6 — セキュリティ (条件付き):**
対象設計が信頼境界を跨ぐデータフロー (外部入力・認証認可・秘密情報・外部公開エンドポイントのいずれか) を含む場合、**aidd:security-reviewer agent** を同じメッセージの並列 dispatch に追加する。含むかどうかはメインループが対象の内容から判定する (キーワード一致ではなく意味で判断)。内部ライブラリ・バッチ処理など信頼境界を跨がない設計では起動しない。`--security` 指定時は判定を省いて必ず起動し、`--no-security` 指定時は起動しない。渡すのは対象のみ (信頼境界の抽出と STRIDE 適用は agent 定義側の手順に従う)。指摘は Agent 1-4 と同じく反証ステージ・裁定に合流する。

Agent 1-4 への指示: 観点ごとに「問題なし」か「指摘 (深刻度: high/mid/low)」を返す。深刻度は次のルーブリックに照合して判定させる (正典は skills/review-loop。変更時は同期):

- high = リリースすると実害が出る (データ破壊・欠損、セキュリティ、本番障害に直結)
- mid = 正しさ・保守性に影響するが、回避策があるか影響範囲が限定的
- low = スタイル・好み・改善アイデア。放置しても実害がない

指摘には必ず具体的な代案を付ける。加えて以下の読了プロトコルを厳守させる:

- 担当ファイルは**全文を読む**。行数制限付きの部分読みは禁止。長いファイルは分割して最後まで読む
- 「未解決」「未定義」「考慮漏れ」系の指摘を出す前に、文書の残りのセクション (特に後半にある決定事項・改定履歴・補足の類) で既に解決済みでないか確認する。解決済みなら指摘しない
- 結果の冒頭に**読了状況**を明記する: 全文読了したファイルと、読み切れなかったファイル (理由付き) を列挙する。読み切れなかったファイルについては指摘を出さず「未レビュー」と報告する

Agent 5 の出力形式は agent 定義側 (確認済み/反証あり/未確認) に従う。

反証ステージ: Agent 1-4 (および起動していれば Agent 6) の結果が揃ったら、arbiter に渡す前に high/mid の指摘を対象と併せて **aidd:refuter agent (sonnet)** に渡し、指摘ごとに反証を試みさせる (指摘が多い場合は分割して並列 dispatch してよい)。セキュリティ関連の指摘には「refuter 定義のセキュリティ反証規則 (攻撃経路が構成不能である現物証拠が必須) に従う」ことを指示に含める。反証成立した指摘は棄却し、棄却理由付きで最終報告に別枠で載せる。low の指摘と Agent 5 の結果 (外部情報源で検証済み) は refuter を通さない。

統合時: 反証を生き延びた指摘・low の指摘・(起動していれば) Agent 5 の結果と対象を **aidd:design-arbiter agent (opus)** に渡して裁定させる。arbiter が矛盾の裁定・深刻度の再ランク付け・「最優先で直すべき1点」の選定を行う。arbiter への指示に以下を含める: 再ランク付けは上記の深刻度ルーブリックに各指摘を独立に照合して行い、指摘の記述の長さや提示順で重み付けしない。読了状況で「読み切れなかった」と申告されたファイルに紐づく指摘は棄却する。「未解決」「未定義」系の指摘は、agent が決定事項・改定履歴の確認を経ていることが読み取れない場合は採用せず、要再確認として別枠で報告する。Agent 5 が起動している場合、arbiter への指示に「source-verifier の『反証あり』のみ深刻度付き指摘として扱い、『未確認』は指摘に含めず情報として併記する」を加える。メインは arbiter の結果をそのまま報告する (メインループのモデルに依存させないため、統合判断を自分で行わないこと)。重大な設計判断でセカンドオピニオンが欲しい場合は codex:rescue に同じ観点を投げて突き合わせる選択肢がある。

## セキュリティ委譲

Agent 6 が担当するのは設計書に対する STRIDE ベースの脅威レビューまで。実コード diff のセキュリティ監査・脆弱性スキャンはスコープ外のため、Agent 6 が high/mid のセキュリティ指摘を出した場合、または `--no-security` で Agent 6 を起動しなかったが設計が信頼境界を跨ぐ場合は、レビュー結果の最後に実装後の `/security-review` の実行を推奨として明記する。
