---
description: 設計を独立マージ可能なPR単位に分割し、GitHub issue 化する
argument-hint: [設計書のパス or 設計の要約]
---

対象: $ARGUMENTS (設計書のパスが渡された場合は全文を読む)

設計された実装を、独立してマージ・テスト可能なPR単位に分割してください。レビュー単位が小さいほどレビュー精度が上がる ([Google Small CLs](https://google.github.io/eng-practices/review/developer/small-cls.html))。セッション内のタスク分解は superpowers:writing-plans の担当であり、このコマンドが扱うのはPR/issue 単位 (チーム・セッション横断の成果物単位) の分割のみ。

**分割の原則**:

- 縦切り (機能スライス) で切る。層ごとの横切り (モデル層だけ・UI層だけ) は単体でマージ・検証できないため禁止
- 各単位は「それ単独でマージしてもプロジェクトが壊れない」こと。依存があるなら順序を明示する
- 目安: 1単位 = 変更5ファイル以内。設計書の「構成と責務」の境界を第一の切り口に使う
- 分割しても1単位で済む規模なら「分割不要」と報告して終了する

**分割案の提示** (issue 作成前に必ずユーザーへ見せる)。各単位について:

1. タイトル (日本語。そのまま issue タイトルになる)
2. スコープ: 含む変更と、触るファイルの見込み
3. 完了条件: 単独でマージ可能と判断できる検証内容
4. 依存: 先行してマージが必要な単位 (なければ「独立」)

**issue 化**: 分割案を AskUserQuestion で確認し (issue を作成する / 分割案のみで終了)、承認された場合のみ `gh issue create` で作成する。タイトル・本文は日本語。本文には設計書のパス・スコープ・完了条件・依存関係を記載する。`gh` が使えない、または GitHub を使っていないプロジェクトでは分割案の提示のみで終了する。
