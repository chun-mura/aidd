# 並列調査フォーメーション

複雑な調査を複数の独立した問いに分解し、scout エージェント群を並列起動して高速に情報を集める。単一メッセージで複数問を一気に委譲し、結論だけ受け取る。

## いつ使うか

未知のコードベース全体を把握する、機能の影響範囲を特定する、技術選定の比較検討材料を集める場合。「独立した問いが複数個ある」調査がユースケース。1問だけなら scout は過剰。依存関係があれば順序立てが必要。

## 構成

**メイン** (判断役) が問いを分解 → **scout ×N** (haiku, 読み取り専用) が各問を調査 → 1メッセージで並列起動 → 結論のみ回収。scout は 30行以内、file:line 参照形式で返す。メインは結果を検証・統合して判断に使う。

## 分解の仕方

**場所で割る**: 大規模リポジトリの `src/api/` と `src/ui/` の依存関係を調べるなら、API側の動作を調査する scout と UI側の呼び出しパターンを調査する scout を独立実行。両者の結果から interface の破損がないか判定。

**観点で割る**: マイグレーション時に「パフォーマンス」「セキュリティ」「互換性」を各 scout で並列調査。複数視点の証拠を同時に集める。

**仮説で割る**: 「バグは X だ」「実は Y だ」を複数仮説として scout に投げ、各仮説に該当するコード片を file:line で並列収集。一番強い仮説を確定。

## プロンプトの型

scout への依頼は以下の要素を必ず含める:
- **対象パス**: 調査範囲 (e.g. `src/hooks/`)
- **問い**: 1文で明確に (e.g. useEffect の副作用パターンを列挙)
- **出力形式**: file:line 参照を要求 (e.g. `filename.ts:45`)
- **行数上限**: 30行以内に厳格制限

例: 「`src/api/client.ts` を読み取り専用で調査し、Promise チェーンと async/await の使い分けパターンを file:line つきで 3つ列挙。30行以内」

## 落とし穴

- 問い同士に依存があるのに並列化する → 前提が不完全で結論が破綻。依存グラフを先に描く。
- scout の報告を検証せず設計判断に使う → エージェントの誤読は発生。重要な判断の前に spot-check。
- 2-3ファイルで済む調査をわざわざ scout に委譲する → メイン側で Read 1-2回で十分。並列化のコストが見合わない。
