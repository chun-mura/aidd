# モデル使い分け (haiku / sonnet / opus / fable)

## いつ使うか

サブエージェント起動時に model パラメータを指定するとき。コスト・速度と品質のトレードオフを判断する。指定しない場合はメインループのモデルを継承。

## 原則

- **検証可能性で選ぶ**: 出力を機械的に検証できるほど下位モデルで足りる
- **判断コストで選ぶ**: 曖昧さ処理や折衝判断が必要なら上位モデルを使う
- **デフォルトは継承**: model 未指定ならメインループのモデルが引き継がれる

## 使い分け表

| タスク | モデル | 理由 |
|--------|--------|------|
| ファイル検索・grep・ログ解析 | haiku | 事実の読み取り、判断不要、正確さはテストで確認 |
| テスト実装・定型コード生成 | sonnet | ロジックは明確、確認可能 |
| 既存設計の解釈・ADRレビュー | sonnet | 中程度の判断、曖昧さ解決は人間に委ねる |
| 新規設計・アーキ判断・トレードオフ | opus/fable | 視点の多面性、長期影響の予測が必要 |
| 要件を踏まえた品質チェック | sonnet | 照合は論理的、要件の解釈は人間承認 |

## アンチパターン

- **設計判断を haiku に委ねる**: スケーラビリティ判断などは背景知識が必須
- **全タスクに opus を使う**: コスト増の割に単純タスクの品質は変わらない
- **速度重視で本来 sonnet なタスクに haiku を使う**: テスト不十分なコードが流出

## このリポジトリでの実例

- `agents/scout.md` → **haiku**: ドキュメント・ログ読み取り、リンク検証など、判断不要の事実収集タスクに特化
- `agents/reviewer.md` → **sonnet**: 要件ドキュメントとの照合、スタイルチェック、論理バグ検出は中程度の判断で足りる
