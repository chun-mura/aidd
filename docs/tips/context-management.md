# コンテキスト管理

## いつ使うか

長いセッションで同じ質問を何度も繰り返す、重要な指示を忘れ始めたとき。大きなタスク開始前の設計フェーズ。応答が劣化した兆候を察知したら実行。

## 原則

- **判断にメインコンテキストを使う**: 収集・検索はサブエージェント (scout等) に並列で出す
- **ファイル全読みは避ける**: 必要な部分のみ Read/grep/リンク参照
- **中間成果物をファイル化**: 大きなJSON、調査結果、設計案は参照渡しにして会話に含めない

## 実践的やり方

- **並列調査**: WebFetch/Bash を並列実行、大きなファイル読み込みは scout に委譲。検索は always を Explore agent に出す
- **compact 前のチェックリスト**: 現タスクの目的、変更ファイル、ブロッカーだけ記録、経過説明は削除。重要な決定や同意は必ず書き出す
- **1セッション 1関心事**: モデル選定と実装は分けセッション、追加要件が出たら新セッション起動。デバッグは別セッション
- **部分読み優先**: `Read ... limit: 50` で冒頭確認、grep で位置把握してからピンポイント読み。全ファイル読みは工程削減の最後の手段

## 兆候と対処

| 兆候 | 対処 |
|------|------|
| 同じ質問の繰り返し / 指示を忘れる | セッション分割、重要指示を CLAUDE.md 化 |
| 応答が長くなる、散文的になる | 目的明確化、タスク縮小、サブエージェント利用 |
| 大きなファイルを何度も読み直している | ファイルを中間ファイルに分割、JSON化して参照 |
| エラー追跡がループしている | 根本原因を separate session で調査 |

## コンテキスト圧縮のチェックリスト

- 現在のタスク目的が1文で言えるか
- 変更したファイルパスが3ファイル以内か（超えたら関連性を再判定）
- ブロッカーが明確に分離できているか

## 実践パターン

**パターン1: 大規模ファイル調査**  
全文読みせず、grep で位置確認 → `limit` 指定で該当箇所のみ Read。複数ファイルなら scout に並列委譲。

**パターン2: マルチステップタスク**  
各ステップ完了後に中間成果物を `.md` ファイルに保存。次のステップはそれを参照渡しで簡潔に。

**パターン3: セッション分割の判断**  
「同じ指示を3回以上繰り返している」or「トークンが70%超過」→ compact / 新セッション起動。
