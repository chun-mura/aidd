# ADR-0001: retro nudge へのサマリ注入 (自動レポート化 scope 外決定の部分変更)

## ステータス

accepted

## 文脈

`usage-log.sh` はローカルの `usage.json` に aidd コマンド利用状況とプロンプト履歴を記録している。一方、20セッションごとの retro nudge は `/aidd:retro` の実行を促すだけで、蓄積済みの情報を利用していない。

既存設計ではフル自動レポート化を scope 外としていた。収集データを軽量に活用しつつ、定期実行や外部サービスを増やさない範囲でその決定を部分的に変更する。

## 決定

20セッションごとの retro nudge 発火時に限り、`usage.json` からコマンド使用回数上位5件と繰り返しプロンプトを最大2件集計して注入する。`jq` が利用できない場合や集計に失敗した場合は、従来の提案文だけを表示する。

## 検討した代替案

### 従来の提案文のみを維持する

- 概要: nudge は `/aidd:retro` の実行提案だけを行う
- 利点: 実装と出力が最小で、依存コマンドを増やさない
- 欠点: すでに収集している利用状況を棚卸し開始時に活かせない
- 不採用理由: 収集済みのローカルデータを要約する限定的な改善に対して、情報不足が大きい

### nudge 時の縮小サマリを注入する

- 概要: 20セッションごとに `jq` でローカルJSONを集計し、10行以内で表示する
- 利点: 実行のきっかけと棚卸し対象を同時に示せる。保存先・収集側・承認フローを変えない
- 欠点: `jq` がない環境ではサマリを表示できない
- 採用理由: `jq` 不在時に従来文へ戻せるため、既存の非ブロッキング運用を維持できる

### cron・ダッシュボードによるフル自動化

- 概要: 定期集計、永続レポート、可視化を自動運用する
- 利点: 継続的な把握と履歴比較ができる
- 欠点: スケジュール管理、保存・表示機構、運用負荷が増える。提示のみ・手動承認という retro の運用境界も曖昧になる
- 不採用理由: フル自動化は引き続き scope 外とし、今回の目的には nudge 時の縮小サマリで十分である

## 影響

- `/aidd:retro` の起動前に、優先的に棚卸しすべきコマンド利用と繰り返しプロンプトを確認できる
- usage.json の書き込み側、retro の手動承認による昇格フロー、外部送信は変わらない
- 出力は10行以内に制限され、`jq` 不在時は情報量が従来どおりになる
